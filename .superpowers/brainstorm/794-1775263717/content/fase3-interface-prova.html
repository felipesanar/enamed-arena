Análise detalhada dos problemas
Crítico · Descoberta
Eliminar alternativa — funcionalidade invisível
O botão "✕" para eliminar uma alternativa é minúsculo, sem label, aparece discretamente na lateral direita de cada opção. Em testes de usabilidade, funcionalidades sem label clara têm taxa de descoberta próxima de zero. Muitos usuários nunca saberão que podem eliminar alternativas.
Impacto
Descoberta
Recomendação: Adicionar um tooltip no hover "Eliminar esta alternativa" + um ícone com label "Eliminar" ao selecionar a linha. Considerar um tutorial overlay na primeira prova destacando a funcionalidade. A eliminação é uma das ferramentas mais valiosas em prova — ela precisa ser descoberta.
Crítico · Compreensão
"Alta confiança" — sem contexto de uso
O botão ⚡ "Alta confiança" existe mas o usuário não tem como saber o que ele faz. Ele aparece no modal de confirmação da prova? Gera alguma análise nos resultados? Impacta o ranking? A funcionalidade existe mas o usuário não tem incentivo para usá-la por não entender o valor.
Confusão
Adoção
Recomendação: Tooltip ao passar o mouse: "Marque quando tiver certeza — aparecerá nos seus resultados para análise de calibração". Mostrar no modal de finalização quantas questões foram marcadas com alta confiança e o percentual de acerto nelas.
Crítico · Affordance
Atalhos de teclado invisíveis na interface
Existem atalhos poderosos: 1-5 seleciona opções, ←→ navega questões, R marca revisão, H marca alta confiança, Esc abre modal de finalização. Mas nenhum desses atalhos está visível na interface. Usuários power-users que poderiam economizar minutos nunca descobrirão.
Eficiência
Recomendação: Adicionar um ícone de teclado no header que expande um painel de atalhos. Alternativamente, mostrar um overlay de atalhos na primeira questão da primeira prova (descartável com "Entendi"). Labels inline sutis: "B → tecle 2" próximo a cada opção no primeiro uso.
Crítico · Risco de erro
Botão "Finalizar" — posição de alto risco no header
O botão "Finalizar" fica no header ao lado do timer. Em mobile, a densidade do header é alta. Um usuário tentando ver o tempo restante pode acidentalmente tocar em "Finalizar". O ícone Esc também abre o modal de finalização — dois caminhos acidentais para encerrar a prova.
Risco
Recomendação: Mover "Finalizar" para a área inferior das questões (junto com Anterior/Próxima). No header, manter apenas informação (título, progresso, timer). Exige confirmação dupla — o modal de confirmação já existe, o risco é chegar nele sem querer.
Atenção · Mobile
Navegador de questões escondido em sheet no mobile
No desktop o navegador de questões fica visível na sidebar. No mobile ele fica escondido num sheet modal que precisa ser ativado. Isso significa que no mobile o usuário não tem visão geral do progresso a menos que abra explicitamente o sheet — gerando desorientação e mais interações.
Mobile UX
Recomendação: Adicionar um mini-navegador fixo no rodapé do mobile: uma barra horizontal com bolinhas coloridas das últimas/próximas questões, swipeable. Mantém a consciência de progresso sem abrir modal.
Atenção · Legenda
Cores do navegador sem legenda visível
O navegador de questões usa 4 cores: branco (não respondida), azul (respondida), amarelo (revisão), verde (alta confiança). Nenhuma dessas cores tem uma legenda visível. O usuário aprende por tentativa e erro ou não aprende.
Clareza
Recomendação: Adicionar uma mini legenda abaixo do grid: "● Respondida ● Revisão ● Alta conf." com os respectivos ícones coloridos. Ocupa 12px de altura e elimina toda a confusão.
Atenção · Tensão
Fullscreen forçado sem aviso prévio claro
A prova entra em fullscreen automaticamente ao iniciar. Isso pode ser inesperado e assustador para usuários que não estão familiarizados. O sistema operacional mostra a notificação padrão do browser ("Pressione Esc para sair do fullscreen"), mas não há aviso contextual do produto explicando o porquê.
Surpresa
Recomendação: Incluir no checklist pré-prova: "A prova abre em tela cheia para sua concentração. Sair do fullscreen é registrado." Isso transforma surpresa em expectativa gerenciada e reforça o propósito da funcionalidade.
Acerto · Manter
Persistência resiliente (localStorage + Supabase)
A estratégia de write-through com localStorage imediato + sync Supabase com debounce de 2s + flush no beforeunload é excelente. O usuário dificilmente perderá progresso mesmo com falha de rede. Isso reduz ansiedade durante a prova.
Manter e comunicar: Adicionar um indicador sutil de "Salvando..." ou um checkmark "Salvo" no header para o usuário saber que o progresso está sendo persistido. Reduz ainda mais a ansiedade sobre perder respostas.
Acerto · Manter
Timer com mudança de estado nos momentos críticos
A mudança de cor do timer em 5 min (warning) e 1 min (destructive + pulse) é boa UX — cria urgência progressiva sem alarmar o usuário desde o início. O estado neutro no começo é correto.
Manter. Considerar adicionar um estado intermediário em 15 minutos restantes — uma cor "atenção suave" antes do warning. O salto de neutro → alarme pode ser brusco.
Diagnóstico Central — Interface da Prova

A interface tem funcionalidades poderosas e subutilizadas. Eliminar alternativas, alta confiança e atalhos de teclado existem no código mas são funcionalmente invisíveis. Um usuário pode fazer 100 questões sem nunca descobrir essas ferramentas. O problema não é ausência de funcionalidade — é ausência de discoverability. A máxima de UX se aplica: uma funcionalidade que não é descoberta não existe para o usuário.
